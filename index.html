Billig lokal förgrening (branching)

Antagligen är Gits mest övertygande egenskap dess flexibla hantering av grenar, som skiljer Git från nästan alla andra versionhanteringssystem (SCM). Grenar i Git är helt annorlunda än i alla andra system som jämförs här, där de flesta syftar till att grenar ska vara en kopia av hela arkivet i en egen mapp.
Git fungerar inte på detta sätt. Git tillåter dig att ha ett flertal lokala grenar som kan vara helt oberoende av varandra och att skapa, slå ihop och ta bort dem tar sekunder.
Detta betyder att du kan göra saker som att
  • skapa en gren för att prova en idé du har, checka in ändringar hur mycket du vill, gå tillbaka till där du var innan, verkställa en patch, gå tillbaka till ditt experiment och till sist slå ihop grenen.
  • ha en gren som alltid innehåller det som är produktionssatt, en annan där du har det som skall testas och flera mindre för dagligt arbete.
  • skapa en ny gren för varje ny funktionalitet du arbetar på så du kan sömlöst hoppa mellan dem, och sedan ta bort grenar som redan slagits ihop med din huvudgren.
  • skapa en gren att experimentera i, inse att det inte ledde till något konkret och sedan ta bort den och överge ditt arbete – utan att någon annan har fått se den, trots att du kan ha tryckt ut ändringar i andra grenar under tiden.
Viktigt att förstå är att när du trycker ut dina ändringar till ett externt arkiv så måste du inte också trycka ut alla dina grenar. Du kan bara dela med dig av en gren, inte alla. Detta brukar ge många möjlighet att experimentera med saker utan att direkt behöva oroa sig för hur och när de ska dela med sig av experimentet till andra.
Du kan komma på sätt att göra detta med några av de andra systemen, men arbetet som skulle vara associerat med dem är mycket svårare och kan gå fel lättare. Git gör denna process otroligt enkel och det förändrar hur de flesta utvecklare arbetar när de lärt sig detta.

Allt är lokalt

Detta är egentligen ganska sant med alla distribuerade SCM:er, men enligt mina erfarenheter så är det extra sant med Git. Det är lite utöver "fetch", "pull" och "push" som kommunicerar på något sätt med något annat än din egen hårddisk.
Detta gör inte bara det mesta operationer snabbare än vad du kanske förväntar dig, utan gör också att du kan arbeta med saker utan en uppkoppling. Det kanske inte låter som en stor sak, men jag blir alltid förundrad över hur ofta jag faktiskt jobbar frånkopplat. Att kunna förgrena, slå ihop grenar, checka in ändringar och se historik samtidigt som du sitter i ett tåg eller ett flygplan är väldigt produktivt.
Även i Mercurial så kommunicerar vanliga kommandon som "incoming" och "outgoing" med en server, medan du med Git kan göra en "fetch" och sedan ha all data från servern, vilket ger dig möjligheten att hantera kod som du inte har i dina lokala grenar än, trots att du är frånkopplad.
Detta betyder således att det är mycket enkelt att ha kopior av inte bara dina grenar, utan också grenarna som tillhör de andra som jobbar i samma arkiv utan att det stökar till dina egna ändringar och grenar.

Git är snabbt

Git är snabbt. Alla — till och med de mest inbitna fansen av de andra systemen — brukar ge Git denna titel. Med Git sker alla operationer lokalt, vilket ger en klart snabbare körning än SVN och Perforce som båda kräver en anslutning till en server för många kommandon. Men även om man jämför med de andra distribuerade SCM:arna så är Git ganska snabbt.
Ett skäl till detta är antagligen eftersom Git byggdes för arbete på Linuxkärnan, vilket innebär att det fanns behov av att det skulle kunna hantera stora arkiv effektivt redan från första början. Dessutom är Git skrivet i C, vilket minskar på allt extra resursanvändande som är associerat med att ha en applikation skriven i ett mer högnivåspråk. Ett till skäl till att Git är snabbt är att utvecklarna haft detta som ett mål med applikationen och därför utvecklat Git för det.

Förberedelseområdet (Staging)

Olikt de andra systemen så har Git vad som kallas för "Förberedelseområdet" ("staging area"), eller "Registret" ("index"). Detta är ett mellansteg för dina ändringar så du kan ordna till dem så att de ser ut som du vill innan de checkas in.
Det häftiga med förberedelseområdet, och det som separerar Git från de andra verktygen, är att du enkelt kan förbereda några av dina filer när du är klar med dem och checka in dem utan alla andra filer du ändrat i din arbetskatalog, eller behöva lista dina ändrade filer på kommandoraden då du checkar in dem.
Allt detta betyder också att du kan förbereda delar av en modifierad fil. Borta är dagarna där det är ett problem med att du gjort två orelaterade ändringar i samma fil utan att checka in den första innan du började på den andra. Med Git kan du bara förbereda ena ändringen, checka in den och sedan förbereda den andra ändringen. Detta skalar till hur många olika ändringar du behöver!
Självklart är det enkelt att ignorera denna funktionalitet om du inte vill ha sådan kontroll — bara släng på '-a' till ditt incheckningskommando så kommer Git att förbereda alla ändrade filer och sedan checka in dem.

Distribuerad

En av de häftigaste finesserna med alla de distribuerade versionshanterarna, Git inkluderad, är att de är distribuerade. Det betyder att istället för att göra en utcheckning av ett arkiv så klonar du det.
En positiv konsekvens av detta är att även om ni arbetar från ett centraliserat flöde så kommer alla utvecklare att ha en så gott som full backup av huvudservern utifall servern råkar ut för ett fel eller än korruption. Det finns alltid redundans så länge det inte bara finns en utvecklare på en dator.
Allt detta saktar inte ned saker och ting mycket heller. I snitt var en utcheckning i SVN endast marginellt snabbare än någon annan av alla distribuerade versionshanterarna. Bland alla DSCM:er som jag testade var Git det snabbaste verktygen.

Valfritt arbetsflöde

En av de fantastiska sakerna med Git är att eftersom det är distribuerat och har ett så överlägset bra greningssystem så kan man använda Git med i stort sett vilket arbetsflöde man vill utan problem.

Subversion-liknande flöde

Ett väldigt vanligt flöde för Git speciellt bland de som konverterat från ett centraliserat SCM, är ett centraliserat arbetsflöde. Git tillåter inte att du trycker upp ändringar om någon annan gjort så sen efter du uppdaterade senast, så en modell där alla trycker upp ändringar till ett och samma ställe fungerar utmärkt.

Flöde med Integrationsledare

Ett annat flöde som är vanligt med Git är det med en integrationsledare — en person som checkar in ändringar i det "välsignade" arkivet, och sedan flera utvecklare som clonar det arkivet och arbetar på egna versioner av det, för att sedan be ledaren att hämta in ändringar från dessa. Detta flöde ser man oftst hos fri och öppen mjukvara och hos GitHub.

Diktator och Löjtnanter

För mer massiva projekt kan man arbeta efter samma flöde som används av Linux- kerneln, där ett flertal personer (de så kallade "Löjtnanterna") är ansvariga för egna specifika delar av projektet och sedan tar in alla ändringar som rör den del som de ansvarar över. Sedan kan en annan ledare ("Diktatorn") ta de ändringar som dennes löjtnanter godkänt och sammanfläta i det "välsignade" arkivet. Alla övriga utvecklare klonar det väsignade arkivet när de ska arbeta på något.

Git är väldigt flexibelt när det kommer till detta, så du kan blanda och skapa dina egna arbetsflöden. Det är inget specifikt inom Git som avgör hur du skall arbeta med din kod.

GitHub

Jag kan vara partisk här, eftersom jag är anställd hos GitHub, men jag la till denna sektion ändå eftersom så många människor säger att GitHub var ett specifikt skäl till att de började använda Git.
GitHub är ett skäl till att använda Git för många människor eftersom det är mer som ett socialt nätverk för kod än bara ett webbhotell. Folk hittar projekt och kodbaser som liknar det de själva håller på med och kan enkelt skapa förgreningar av de projekten och bidra med egna ändringar. Detta skapar en stor närskap runt Git och projekten som finns här.
Det finns andra tjänster, både för Git och för de andra SCM:er, men få är så användarinriktade med en sådan social fokus, och ingen av dem har ens i närheten så många användare. Den sociala aspekten är störtskön, och kombinationen av de ovan nämnda finesserna gör att utveckling med Git och GitHub är en mycket bra kombination för att snabbt utveckla öppen mjukvara.
Denna typ av community är helt enkelt inte ens tillgänglig med någon annan SCM.

Lätt att lära sig

Detta är givetvis mycket svårt att bevisa objektivt utan någon form av studie, så jag kommer bara att visa skillnaden mellan help-kommandot för Git och Mercurial.
